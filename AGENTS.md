# AGENTS.md

기준 날짜 및 시간: 2026-04-13 18:48:26 (Asia/Seoul)

## 1. 목적
이 문서는 이 저장소에서 작업하는 에이전트가 현재 프로젝트 구조와 도메인 규칙을 깨뜨리지 않고 수정, 분석, 문서화 작업을 수행하도록 돕는 작업 규칙 문서다.

실제 백엔드 프로젝트는 `Capstone/` 하위에 있다. 문서는 저장소 루트와 `docs/` 하위에 있다.

## 2. 현재 프로젝트 상태 해석
현재 저장소는 아래 상태로 이해한다.
- Spring Boot 백엔드 구조가 이미 존재한다.
- `Controller / Service / Repository / Domain / DTO` 계층이 이미 잡혀 있다.
- OAuth2 + JWT 인증 흐름이 이미 존재한다.
- 리스트와 리스트-식당 평가가 핵심 도메인이다.
- 일부 정책은 코드와 테스트로 강제되고, 일부는 추가 확인이 필요하다.

즉, 이 프로젝트는 새 구조를 만드는 단계가 아니라 기존 구조를 유지하며 정책과 구현을 정리하는 단계다.

## 3. 작업 시작 전 읽기 순서
1. `GUIDE.md`
2. `AGENTS.md`
3. `DB.md`
4. 필요한 `docs/db/*`
5. `LOGIC.md`
6. 필요한 `docs/logic/*`

8. `docs/current-gaps.md`

## 4. 가장 중요한 원칙
### 기존 구조를 최대한 유지한다
- 계층 구조를 함부로 바꾸지 않는다.
- 인증 구조를 전면 재작성하지 않는다.
- 엔티티 전체를 갈아엎지 않는다.

### 도메인 규칙을 먼저 본다
- 기능 추가보다 먼저 `DB.md`, `LOGIC.md`, 관련 하위 문서를 확인한다.
- 리스트, 점수, 공개/비공개, 대표 리스트, soft delete는 특히 신중히 다룬다.

### 작은 단위로 수정한다
- 문제 지점을 찾고
- 영향 범위를 확인하고
- 필요한 부분만 최소 수정한다

### 모르는 내용은 추정하지 않는다
- 근거 없는 엔드포인트 추가 금지
- 근거 없는 규칙 확정 금지
- 불확실하면 `추가 확인 필요`로 남긴다

### 설계 목표와 현재 구현을 섞지 않는다
문서를 읽거나 수정할 때 아래를 구분한다.
- 설계상 목표 또는 해석
- 현재 코드 기준 동작
- 추가 확인 필요 사항

## 5. 금지사항
### 구조 재작성 금지
- 전체 아키텍처 재설계 금지
- 계층 구조 임의 변경 금지
- 인증 흐름 전면 교체 금지

### 도메인 규칙 임의 변경 금지
- 대표 리스트 의미 임의 변경 금지
- 공개/비공개 의미 임의 변경 금지
- `autoScore` 계산식 임의 변경 금지
- `isDeleted`, `isHidden` 의미 임의 변경 금지

### 문서 무시 금지
- 정책/구조 변경이 있으면 관련 문서도 함께 검토한다.
- 허브 문서와 세부 문서의 역할을 섞지 않는다.
- 코드 변경만 있고 문서 반영이 없는 상태를 허용하지 않는다.

## 6. 문서와 코드가 다를 때의 처리 방식
문서와 코드가 다를 때는 아래처럼 처리한다.
- 문서 허브에는 세부 정책을 억지로 넣지 않는다.
- 세부 문서에는 `현재 코드 기준`, `추가 확인 필요`, `후속 수정 후보`를 분리해서 적는다.
- 차이를 발견했다고 해서 임의로 새 규칙을 만들지 않는다.
- 명확한 근거가 없으면 정책을 확정하지 않는다.
- 이미 코드와 테스트로 강제된 규칙은 확정된 정책으로 문서에 반영한다.
- 아직 확정되지 않은 항목만 `추가 확인 필요` 또는 `docs/current-gaps.md`로 남긴다.

## 7. 프론트 강제 정책을 다루는 태도
프론트/백엔드/DB 중 어디에서 강제할지 열어둘 수 있는 항목은 아직 정책이 확정되지 않았거나 구현되지 않은 항목뿐이다.

현재 하네스 기준으로 아래 규칙은 백엔드 검증 기준으로 확정된 상태로 본다.
- 리스트 최소 5개 규칙
- 동일 리스트 내 중복 식당 금지
- 리스트와 식당 지역 exact match

따라서 위 규칙들에 대해서는 `프론트에서 처리할 수도 있다`는 식의 표현을 사용하지 않는다.

아직 확정되지 않은 항목만 아래처럼 구분한다.
- 현재 백엔드 코드와 테스트로 강제되는 정책
- 아직 정책 확정이 필요한 항목
- 구현 계층(프론트/백엔드/DB) 결정이 남은 항목

미확정 항목은 `추가 확인 필요` 또는 `docs/current-gaps.md`에 남긴다.

## 8. 현재 프로젝트에서 특히 주의할 영역
- `UserList` / `ListRestaurant` 관련 로직
- `autoScore` 계산식
- 대표 리스트 정책
- 공개 / 비공개 정책
- soft delete / hide 필터
- OAuth2 로그인 후 JWT / Refresh Token 흐름

## 9. 문서 수정 원칙
- `GUIDE.md`는 문서 허브로 유지한다.
- 저장 구조는 `DB.md`와 `docs/db/*`에 둔다.
- 동작 흐름과 정책은 `LOGIC.md`와 `docs/logic/*`에 둔다.
- 주요 차이와 미정 항목은 `docs/current-gaps.md`에 모은다.

### 필수 문서 업데이트 규칙
- 작업 중 확정되거나 변경된 정책, 계산식, 해석 기준은 반드시 문서에 반영한다.
- 확정된 정책은 `docs/logic/*` 또는 해당 정책 문서에 기록한다.
- 아직 확정되지 않은 항목은 `docs/current-gaps.md`에 기록한다.
- 문서와 코드/설계 간 불일치를 남기지 않는다.
- 코드 변경만 있고 문서 반영이 없는 상태를 허용하지 않는다.
- 운영 문서는 Git에서 추적되는 상태로 유지한다.
- `validation-rules.md`는 공통 기반 검증만, `current-gaps.md`는 unresolved only로 유지한다.

## 10. 최종 목표
이 저장소에서 에이전트의 목표는 새 구조를 발명하는 것이 아니다.

목표:
- 현재 코드 구조 정확히 이해
- 문서와 코드의 차이 명확히 구분
- 필요한 최소 수정 수행
- 확장 가능성을 해치지 않는 보완
- 문서와 코드의 일관성 유지

## 11. 실제 작업 루트와 실행 기준
- 저장소 루트와 실제 백엔드 프로젝트 루트는 다를 수 있다.
- 백엔드 코드 작업은 기본적으로 `Capstone/` 하위에서 수행한다고 가정한다.
- 루트 문서 수정과 백엔드 코드 수정의 경로를 혼동하지 않는다.

### 기본 확인 순서
1. 현재 작업 대상이 문서인지 코드인지 먼저 확인한다.
2. 코드 작업이면 `Capstone/` 기준으로 파일을 찾는다.
3. 수정 전 관련 문서(`DB.md`, `LOGIC.md`, `docs/current-gaps.md`)를 확인한다.
4. 수정 후 관련 문서 갱신 필요 여부를 판단한다.

## 12. 작업 완료 조건
아래 조건을 만족해야 작업 완료로 본다.
- 요청 범위를 벗어난 수정이 없다.
- 관련 코드와 문서가 서로 모순되지 않는다.
- 불확실한 내용은 확정하지 않고 `추가 확인 필요`로 남긴다.
- 문서와 코드의 차이를 발견했으면 `docs/current-gaps.md` 반영 여부를 검토한다.
- 확정된 정책, 계산식, 해석 기준은 관련 문서에 반영되어 있다.

## 13. 수정 후 검증 원칙
수정 후 아래 순서를 기본으로 따른다.
1. 변경 영향 범위를 다시 확인한다.
2. 관련 로직의 입력/출력/검증 조건을 점검한다.
3. 실행 가능한 테스트가 있으면 우선 실행한다.
4. 테스트가 없으면 수동 검증 포인트를 명시한다.
5. 실패하거나 불확실하면 완료 처리하지 말고 원인을 남긴다.

`autoScore` 계산식, 리스트 정렬, 추천/랭킹 관련 작업 전후에는 `ListRestaurantAutoScoreTest` 실행 여부를 먼저 검토한다.

## 14. 현재 MVP 단계에서 특히 확인할 입력 데이터 규칙
추천/랭킹이 아직 미구현이어도, 이후 기능의 기반이 되는 아래 규칙은 중요하게 본다.
- 리스트 최소 5개 규칙
- 동일 리스트 내 중복 식당 금지
- 리스트와 식당 지역 exact match
- 점수 필드와 `autoScore` 계산 일관성
- `isDeleted`, `isHidden` 필터 적용 여부
- 공개/비공개가 노출 제어인지 계산 제외인지의 구분

## 15. 테스트/실행 명령은 확인 후만 확정
- 실행 명령, 테스트 명령, 빌드 명령은 현재 코드베이스에서 실제 존재 여부를 확인한 뒤 문서화한다.
- 확인되지 않은 명령을 추정으로 추가하지 않는다.

## 15-1. 검증 운영 방식
- 공통 기반 검증은 프로젝트 전반에서 반복 재사용하는 베이스라인 검증이다.
- 작업별 검증은 현재 수행하는 기능 작업에 따라 달라지는 검증이다.
- 구현 작업 전에는 이번 작업에 필요한 검증 항목을 먼저 정리한다.
- 구현 후에는 공통 기반 검증과 작업별 검증을 함께 점검한다.
- 추천, 랭킹, 리스트 정렬처럼 점수 해석에 영향을 주는 작업은 공통 기반 검증부터 먼저 확인한다.

## 16. Plan-First Workflow
복잡한 작업은 구현 전에 먼저 계획 문서를 작성한다.

계획 문서를 먼저 작성해야 하는 대표 작업:
- 복잡한 기능 추가
- 추천 / 랭킹 / 배치 설계
- 구조 영향이 큰 수정
- 다단계 작업
- 반나절 이상 걸릴 수 있는 작업

기본 원칙:
- 계획 문서는 저장소 루트의 `PLANS.md` 템플릿을 따른다.
- 실제 작업별 계획은 `PLANS.md` 형식을 복사해서 작성한다.
- 계획 작성 후 바로 구현하지 않는다.
- 사용자 검토 또는 승인을 받기 전까지는 분석과 문서화까지만 수행한다.
- 승인 후 구현을 시작한다.
- 진행 중에는 계획 문서의 `Progress`를 계속 갱신한다.
- 작업 완료 후에는 검증 결과와 남은 이슈를 계획 문서 또는 `docs/current-gaps.md`에 반영한다.

주의:
- 계획 문서 없이 복잡한 구현을 바로 시작하지 않는다.
- 계획 문서는 고정 문서가 아니라 진행에 따라 갱신하는 living document로 다룬다.
