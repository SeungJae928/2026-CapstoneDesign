## 1. 문제 해결에 적용된 이론
이 프로젝트는 일반적인 리뷰 텍스트 수집형 맛집 서비스가 아니라, 사용자가 직접 만든 리스트와 리스트 안의 식당 평가를 핵심 데이터로 삼는 구조를 선택했다. 이렇게 설계한 이유는 추천과 랭킹의 입력을 자유서술형 텍스트가 아니라 구조화된 점수와 관계 데이터로 확보하기 위해서다. 실제 구현에서는 `UserList`가 사용자의 취향 맥락을 담는 단위가 되고, `ListRestaurant`가 특정 식당에 대한 명시적 평가를 저장하는 역할을 맡는다. 이 방식은 “어떤 식당을 좋아했는가”뿐 아니라 “어떤 지역 맥락에서, 어떤 리스트 안에서, 어떤 기준으로 평가했는가”를 함께 남길 수 있어 이후 추천 로직을 확장하기에 유리하다.

점수 모델은 다기준 평가 개념을 따른다. 현재 구현에서는 맛(`taste`), 가성비(`value`), 분위기(`mood`)를 각각 입력받고, 이를 하나의 종합 점수인 `autoScore`로 변환한다.

```text
autoScore = (taste * 0.6 + value * 0.2 + mood * 0.2) * 10
```

이 공식을 선택한 이유는 맛을 가장 중요한 선호 축으로 두되, 가성비와 분위기도 무시하지 않기 위해서다. 최종적으로 100점 스케일로 저장하도록 한 것은 랭킹과 추천에서 점수를 직관적으로 해석하고, 이후 정규화(`score / 100.0`)하여 재사용하기 쉽게 만들기 위함이다.

랭킹 설계에는 소수 표본 편향을 줄이기 위한 베이지안 평균 개념을 적용했다. 현재 구현은 식당별 평균 점수 `R`, 평가 수 `v`, 전체 평균 `C`, 보정 상수 `m=5`를 사용해 아래 형태의 보정 점수를 계산한다.

```text
adjustedScore = (v / (v + m)) * R + (m / (v + m)) * C
```

또한 같은 사용자가 같은 식당을 여러 리스트에 담았더라도, 랭킹과 추천 계산에서는 `user_id + restaurant_id` 기준 최고 `autoScore` 한 번만 반영하도록 해 동일 사용자의 중복 행동이 과도하게 집계를 왜곡하지 않게 했다.

추천 설계에는 콘텐츠 기반 추천과 협업 필터링을 결합한 하이브리드 접근을 적용했다. 식당 추천에서는 카테고리 집합 유사도, 카테고리 선호 분포 적합도, 랭킹 보정 점수, 이웃 사용자 유사도 기반 협업 신호, 지역 적합도를 함께 사용한다. 리스트 추천에서는 후보 리스트를 하나의 벡터로 해석하고, 식당 일치도, 카테고리 분포, 점수 성향 벡터, 품질 점수, owner similarity, 지역 점수를 조합한다.

지역 우선 전략은 추천 품질을 높이기 위한 도메인 해석에서 나왔다. 현재 구현은 식당 추천과 리스트 추천 모두에서 먼저 같은 지역 후보를 모으고, 부족할 때만 다른 지역 후보를 fallback으로 사용한다. fallback 후보에는 더 낮은 `regionScore`를 부여해 우선순위를 조정한다.

## 2. SW를 이용한 설계/해석
소프트웨어 구조는 추천과 랭킹을 나중에 붙이는 방식이 아니라, 처음부터 이를 고려한 저장 구조로 설계됐다. 핵심은 `User - UserList - ListRestaurant - Restaurant`로 이어지는 계층이다. `UserList`는 사용자의 취향 묶음이며, `ListRestaurant`는 식당에 대한 구조화된 평가 기록이다. 이 때문에 추천과 랭킹은 별도의 리뷰 테이블 없이도 `list_restaurants`를 사실상 사용자-식당 상호작용 테이블처럼 사용할 수 있다.

리스트를 지역 단위로 분리한 것도 의도적인 설계다. `UserList`는 `regionName`을 필수로 가지며, 리스트에 식당을 추가할 때는 리스트 지역과 식당 지역이 exact match인지 백엔드에서 검증한다. 이는 서로 다른 생활권 식당이 하나의 취향 단위로 섞여 들어와 추천 프로필을 왜곡하는 것을 막기 위한 선택이다.

점수 계산은 입력 시점에 확정해서 저장하는 방향을 택했다. `ListRestaurant`는 생성 시와 점수 수정 시 모두 같은 계산식으로 `autoScore`를 다시 만든다. 이 방식 덕분에 추천 서비스, 랭킹 서비스, 검증 SQL, 테스트 코드가 모두 동일한 점수 기준을 공유할 수 있다.

랭킹 구조는 `user_id + restaurant_id` 기준으로 최고 `autoScore`만 남긴 뒤, 그 결과를 식당별로 집계한다. 이후 단순 평균 대신 베이지안 보정 점수를 적용해 표본이 적은 식당의 과대평가를 줄인다. 이 과정은 `RestaurantRankingRepositoryImpl`의 native SQL에 직접 반영되어 있다.

식당 추천은 사용자 프로필을 먼저 만들고 후보 식당을 평가하는 2단계 구조로 설계됐다. 현재 구현은 활성 리스트 전체에서 사용자의 `restaurantBestScore`, `categoryPreference`, `regionPreference`, `topLikedRestaurants`, `dominantRegion`을 계산한다. 이후 같은 지역 식당 후보를 우선 조회하고, 필요할 때만 fallback 지역 후보를 추가한다.

리스트 추천은 “다른 사용자의 공개 리스트” 자체를 추천 대상으로 본다. 그래서 현재 사용자의 활성 리스트들로 취향 프로필을 만들되, 후보는 다른 사용자의 `isPublic = true` 리스트만 사용한다. 후보 리스트는 식당 구성, 카테고리 분포, 평균 점수 스타일, 리스트 평균 품질, owner similarity 같은 feature로 해석되고, same-region 결과를 먼저 채운 뒤 부족할 때만 fallback을 뒤에 붙인다.

입력 데이터 보호를 위해 일부 도메인 규칙은 백엔드 공통 검증으로 승격했다. 현재 백엔드는 리스트 최소 5개 규칙, 동일 리스트 내 중복 식당 금지, 리스트-식당 지역 exact match를 실제 서비스 로직과 테스트로 강제한다.
