# 통합 검색 정책

## 기본 구조

검색 화면은 하나의 검색어를 받아 아래 3개 탭으로 나눠 보여준다.

- 맛집
- 유저
- 지역

백엔드 API는 `GET /search?query=...` 하나를 사용한다.
응답은 `restaurants`, `users`, `regions`를 동시에 내려주고, `primaryType`으로 우선 노출 탭을 알려준다.

## 맛집 검색

맛집 탭은 식당 이름만 검색하지 않는다.
아래 필드를 검색 대상으로 본다.

- 식당명
- 주소/도로명주소
- 지역명
- 카테고리
- 상위 카테고리
- 메뉴명
- 정규화 메뉴명
- 태그명

예시:

- `와이앤웍`: 상호 검색
- `한식`: 카테고리 검색
- `돈까스`: 메뉴/태그/상호 검색
- `용인 돈까스`: 지역 필터 + 음식 검색

내부 DB 식당이 없고 유저 검색 결과도 없으며 generic browse 검색이 아닐 때만 네이버 PC Map fallback 식당을 섞는다.

## 유저 검색

유저 탭은 닉네임 기준으로 검색한다.

- `@tester`: 명시적 유저 검색
- `tester`: 일반 검색어지만 닉네임과 매칭되면 유저 탭에 노출

일반 닉네임 검색에서 유저 결과가 있으면 외부 식당 fallback을 섞지 않는다.
유저 검색 결과가 식당 외부 fallback에 묻히는 것을 막기 위함이다.

## 지역 검색

지역 탭은 검색어가 실제 지역 신호와 매칭될 때만 채운다.
지역 탭은 지역 자체 상세가 아니라 메인 랭킹 진입점이다.

예시:

- `용인`
- `처인구`
- `역북동`
- `용인 맛집`

지역 결과의 `rankingPath`는 현재 `/rankings/restaurants?regionName=...` 형태다.
읍/면/동 검색어가 들어와도 현재 랭킹 API는 `regionName` 기준으로 연결된다.

## 현재 위치 기준 지역 랭킹

현재 `GET /search`는 위치 정보를 받지 않는다.
따라서 검색어가 지역이 아닐 때 지역 탭을 현재 위치 기반 랭킹으로 채우지 않는다.

현재 위치 기반 랭킹이 필요하면 아래 중 하나가 별도 계약으로 필요하다.

- 클라이언트가 현재 위치를 지역명으로 변환한 뒤 `/rankings/restaurants?regionName=...` 호출
- `/search`에 `currentRegionName` 같은 선택 파라미터 추가
- 좌표 기반 지역 resolver 추가 후 랭킹 API와 연결

현재 단계에서는 검색 API가 임의로 현재 위치 랭킹을 추론하지 않는다.

## 제외

- 검색과 리스트 추가 검색을 통합하지 않는다.
- 지역 탭을 검색어와 무관한 기본 추천 영역으로 사용하지 않는다.
- 유저 닉네임 검색에 외부 식당 fallback을 우선 노출하지 않는다.
